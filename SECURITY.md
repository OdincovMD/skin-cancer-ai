# Security Policy

Спасибо, что помогаете делать проект безопаснее.

## Supported Scope

Особенно важны сообщения о проблемах в следующих зонах:

- аутентификация и JWT
- API-ключи и их хранение
- callback-авторизация между сервисами
- загрузка и выдача изображений
- хранение файлов в MinIO
- email verification и password reset
- утечки секретов, персональных данных или медицинских данных

## How To Report

Пожалуйста, не публикуйте уязвимости в публичных issues.

Для приватного сообщения подготовьте:

1. краткое описание проблемы
2. шаги воспроизведения
3. ожидаемое и фактическое поведение
4. оценку возможного влияния
5. если возможно, безопасный proof of concept без реальных пользовательских данных

Если в вашей среде уже используется отдельный security email или private intake channel, укажите его здесь перед публичной публикацией репозитория.

В текущем состоянии репозитория безопаснее всего заменить этот абзац на конкретный приватный контакт владельца проекта перед open-source публикацией.

## Disclosure Guidelines

- Не прикладывайте реальные секреты, токены, `.env` или дампы прод-данных.
- Не публикуйте персональные данные пациентов, врачей или пользователей.
- Не проводите разрушительные тесты на чужой инфраструктуре.

## Expected Response

Целевой процесс:

1. подтверждение получения отчёта
2. первичная triage-оценка
3. согласование серьёзности и плана исправления
4. выпуск исправления
5. согласованное публичное раскрытие, если это уместно

Если будете публиковать репозиторий широко, лучше дополнить этот файл:

- конкретным security contact
- ожидаемыми SLA по ответу
- supported versions / branches
